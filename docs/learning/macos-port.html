The native macOS port

System map

AsterionMac is an independent SwiftPM GUI app targeting macOS 26. The iOS target remains unchanged. The desktop app talks directly to the same production content and user API and uses Clerk's macOS-capable Swift SDK for authentication.

Responsibility and ownership

  • AsterionMacApp owns app-wide services and declares main, reader, and settings scenes.
  • AsterionMacApp gives Clerk an app-owned Keychain service, while the staging script gives every build the same signed application identity.
  • AsterionAppDelegate is the only AppKit bridge; it activates the staged SwiftPM app bundle and brings its SwiftUI window forward.
  • ContentView owns window-scoped sidebar and novel selection with @SceneStorage.
  • AppModel owns the catalog, Clerk session, server library, chapter cache, independent per-novel offline download state, and progress reconciliation.
  • OfflineLibraryStore owns downloaded novels and full chapter files under Application Support.
  • ReadingProgressStore owns account-scoped local positions and pending sync revisions under Application Support.
  • APIClient is an actor, so token and request state are serialized away from view code.
  • ReaderView owns each reader window's active chapter, chapter navigator visibility, visible line, export, and preference bindings.

Desktop decisions

The mobile tab bar and push stacks were replaced with a stable three-column NavigationSplitView. A novel stays selected while the user moves between the list and detail surface. The detail column has a 640-point floor and the workspace minimum includes all three column floors, preventing split-view restoration or resizing from compressing the novel detail below its designed measure. Reading opens in a value-driven WindowGroup, allowing multiple books or chapters to remain open without hiding catalog context. Reader windows use a Chicken-style shell with in-app chrome, a centered chapter readout, a slim progress bar, and Paper, Sepia, and Ink themes that style both the text surface and surrounding chrome. The selected theme is an app preference shared by reader windows. Facing pages appear at desktop width; turning past a chapter edge opens the neighboring chapter so the book keeps moving forward or backward. Narrow windows keep a single page so the text never feels compressed. Downloaded novels become local library items and can be opened without a network session. Account preferences live in the system Settings scene, while account identity remains an app section.

Behavior walkthrough

  1. Launch starts catalog loading and Clerk session restoration concurrently.
  2. Sidebar selection chooses a catalog projection: alphabetical, ranked, or saved.
  3. Selecting a novel renders its metadata and lazily fetches its complete chapter list.
  4. Read opens a new reader window using the novel and chapter identifiers as the route.
  5. Download fetches chapter bodies through a bounded 32-request task group, preserves chapter order before storage, and writes one JSON package per novel to the offline library.
  6. Each download publishes its own completed and total chapter counts; the selected novel shows inline progress while the sidebar download center shows every active, completed, or failed novel.
  7. Catalog and chapter lookups merge downloaded data with live API data, so downloaded books remain readable offline.
  8. The reader lays chapter text into a WebKit-powered two-page spread when the window is wide enough.
  9. The chapter-list button reveals a searchable, theme-aware navigator that highlights and scrolls to the current chapter; selecting a result cancels any older chapter load before opening the new one.
  10. The theme menu applies Paper, Sepia, or Ink colors to SwiftUI chrome and WebKit content from one palette and persists the selection with @AppStorage.
  11. The reader suppresses both document-level scrollbar styling and nested AppKit scroller views while wheel, keyboard, and trackpad page turns remain active.
  12. The page-turn bridge computes spread width from the viewport instead of parsing CSS calc() or clamp() strings.
  13. At the first or last spread of a chapter, page-turn input loads the previous or next chapter.
  14. The reader debounces visible-line changes, then writes the newest position locally before attempting the server request.
  15. Failed server writes remain marked pending. Authentication events and network reconnection reconcile local and server records by timestamp; a newer local revision uploads, while a newer server revision replaces it.

Keychain identity

Clerk persists its client and environment records in the macOS Keychain. Keychain access is authorized against the executable's designated code requirement, not just the bundle identifier written in Info.plist. The staging script therefore signs the executable with an installed Apple code-signing identity and verifies that it satisfies the Asterion identifier plus Apple's certificate anchor. Clerk uses the explicit service cloud.cyberverse.AsterionMac.clerk, keeping authentication records scoped to this app.

A raw SwiftPM executable only has an ad-hoc, hash-based designated requirement. Copying that executable into an app bundle without re-signing changes its trusted identity whenever the binary changes, which makes macOS request password confirmation for every existing Clerk record. A stable signed requirement removes that authorization loop.

Toolbar ownership

SwiftUI exclusively owns toolbar composition through .searchable, .toolbar(removing:), and scene commands. AppKit notifications must not remove NSToolbarItem instances while SwiftUI is replacing a toolbar: Account navigation changes search availability, and concurrent imperative removal can trigger duplicate KVO teardown inside AppKitWindowController. The app delegate only configures window chrome after launch.

DMG packaging

./script/build_and_run.sh --package builds the release executable, stages the app with version metadata and SwiftPM resource bundles, creates a compressed Asterion.dmg containing the app and an Applications shortcut, signs the image, mounts it read-only, validates the mounted executable and bundle identifier, and emits a SHA-256 checksum. The current machine has an Apple Development identity, so its artifact is suitable for local installation and testing. Public distribution additionally requires a Developer ID Application identity, hardened runtime, a fully sealed app bundle, and notarization.

Verification

swift build proves source and current dependency compatibility. swift test verifies backend decoding, rank normalization, and chapter HTML cleanup. Every staging run verifies the executable's signature and designated Asterion requirement before launch. ./script/build_and_run.sh --verify stages a real .app bundle—including SwiftPM dependency resource bundles—and proves the macOS process launches. The package workflow additionally verifies the DMG filesystem, mounts the image, checks the installed layout, and records the image checksum; the release handoff check launches the app directly from that mounted image.
