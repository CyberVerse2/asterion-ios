The native macOS port

System map

AsterionMac is an independent SwiftPM GUI app targeting macOS 26. The iOS target remains unchanged. The desktop app talks directly to the same production content and user API and uses Clerk's macOS-capable Swift SDK for authentication.

Responsibility and ownership

  • AsterionMacApp owns app-wide services and declares main, reader, and settings scenes.
  • AsterionMacApp gives Clerk an app-owned Keychain service, while the staging script gives every build the same signed application identity.
  • AsterionAppDelegate activates the staged SwiftPM app bundle and brings its SwiftUI window forward. The reader itself is fully SwiftUI, including its WebKit surface.
  • ContentView owns window-scoped sidebar and novel selection with @SceneStorage.
  • AppModel owns the catalog, Clerk session, server library, chapter cache, independent per-novel offline download state, and progress reconciliation.
  • OfflineLibraryStore owns downloaded novels and full chapter files under Application Support.
  • ReadingProgressStore owns account-scoped local positions and pending sync revisions under Application Support.
  • APIClient is an actor, so token and request state are serialized away from view code.
  • ReaderView owns each reader window's active chapter, chapter navigator visibility, visible line, export, and preference bindings.

Desktop decisions

The mobile tab bar and push stacks were replaced with a three-column NavigationSplitView. macOS owns sidebar collapse, column resizing, fullscreen adaptation, restoration, toolbar placement, and catalog search. A novel remains selected while the user moves between the catalog and detail columns. Standard controls receive macOS 26 Liquid Glass automatically, while Asterion's crimson remains the semantic accent and system colors provide adaptive light and dark appearances. Reading opens in a value-driven WindowGroup, allowing multiple reader windows without hiding catalog context. The reader uses a native toolbar for chapter navigation, chapter search, typography, themes, and export. Paper, Sepia, and Ink deliberately control the reading surface independently from the system appearance. Facing pages appear at desktop width; turning past a chapter edge opens the neighboring chapter, while narrow windows use a single scrolling page. Downloaded novels become local library items and remain readable without a network session. Account preferences live in the system Settings scene, while account identity remains a sidebar destination.

Behavior walkthrough

  1. Launch starts catalog loading and Clerk session restoration concurrently.
  2. Sidebar selection chooses a catalog projection: alphabetical, ranked, or saved.
  3. Selecting a novel renders its metadata and lazily fetches its complete chapter list.
  4. Read opens a new reader window using the novel and chapter identifiers as the route.
  5. Download fetches chapter bodies through a bounded 32-request task group, preserves chapter order before storage, and writes one JSON package per novel to the offline library.
  6. Each download publishes its own completed and total chapter counts; the selected novel shows inline progress while the sidebar download center shows every active, completed, or failed novel.
  7. Catalog and chapter lookups merge downloaded data with live API data, so downloaded books remain readable offline.
  8. The reader connects a macOS 26 SwiftUI WebView to a WebPage and loads locally generated chapter HTML for the two-page spread.
  9. The chapter-list button reveals a searchable, theme-aware navigator that highlights and scrolls to the current chapter; selecting a result cancels any older chapter load before opening the new one.
  10. The theme menu applies Paper, Sepia, or Ink colors to SwiftUI chrome and WebKit content from one palette and persists the selection with @AppStorage.
  11. SwiftUI hides the native WebView scroll indicators while document CSS removes web scrollbars; wheel, keyboard, and trackpad page turns remain active.
  12. The page-turn script computes spread width from the viewport instead of parsing CSS calc() or clamp() strings.
  13. A WKUserContentController configured through WebPage.Configuration carries visible-line and chapter-boundary events from the page without an NSViewRepresentable, WKWebView delegate, or AppKit view lifecycle.
  14. At the first or last spread of a chapter, page-turn input loads the previous or next chapter.
  15. The reader debounces visible-line changes, then writes the newest position locally before attempting the server request.
  16. Failed server writes remain marked pending. Authentication events and network reconnection reconcile local and server records by timestamp; a newer local revision uploads, while a newer server revision replaces it.

Keychain identity

Clerk persists its client and environment records in the macOS Keychain. Keychain access is authorized against the executable's designated code requirement, not just the bundle identifier written in Info.plist. The staging script therefore signs the executable with an installed Apple code-signing identity and verifies that it satisfies the Asterion identifier plus Apple's certificate anchor. Clerk uses the explicit service cloud.cyberverse.Asterion.clerk, keeping authentication records scoped to this app.

A raw SwiftPM executable only has an ad-hoc, hash-based designated requirement. Copying that executable into an app bundle without re-signing changes its trusted identity whenever the binary changes, which makes macOS request password confirmation for every existing Clerk record. A stable signed requirement removes that authorization loop.

Toolbar ownership

SwiftUI exclusively owns toolbar composition. NavigationSplitView supplies the native sidebar toggle, .searchable owns catalog search, and ToolbarSpacer separates refresh and download utilities. Reader controls use the same native toolbar model. No AppKit code mutates window chrome or removes toolbar items.

DMG packaging

./script/build_and_run.sh --package builds the release executable, stages the app with version metadata and SwiftPM resource bundles, creates a compressed Asterion.dmg containing the app and an Applications shortcut, signs the image, mounts it read-only, validates the mounted executable and bundle identifier, and emits a SHA-256 checksum. The current machine has an Apple Development identity, so its artifact is suitable for local installation and testing. Public distribution additionally requires a Developer ID Application identity, hardened runtime, a fully sealed app bundle, and notarization.

Verification

swift build proves source and current dependency compatibility. The ten Swift tests verify backend decoding, rank normalization, chapter cleanup, offline persistence, download state, and local-first progress reconciliation. Every staging run verifies the executable's signature and designated Asterion requirement before launch. ./script/build_and_run.sh --verify stages a real .app bundle—including SwiftPM dependency resource bundles—and proves the macOS process launches. The package workflow additionally verifies the DMG filesystem, mounts the image, checks the installed layout, and records the image checksum.
